# 内容
提示：本节内容需要和下一节“对象池思想”一起学习

我相信每个玩过泰拉瑞亚的玩家都会产生这些疑问
那就是这个游戏到底是如何运作的？一秒60个tick，每个tick发生了什么？为什么泰拉瑞亚不能像其他游戏一样调整fps？
本节会揭晓这些疑问的答案。

Q1:为什么泰拉瑞亚不能像其他游戏一样调整FPS？
A:泰拉瑞亚之所以不能像 Unity 游戏那样随意修改 FPS，本质原因在于它的整个游戏逻辑是以“固定 60 帧”为时间单位设计的，而不是以真实时间为单位。对泰拉瑞亚来说，一帧本身就等于 1/60 秒，角色移动、重力、弹幕速度、冷却时间、Buff 持续时间、BOSS 行为切换、AI 计数等几乎所有系统，都是直接按“每帧变化多少”来写的，而不是“每秒变化多少”。因此一旦 FPS 改变，这些基于帧数的计算就会整体加速或减速，游戏节奏立即失真。相比之下，Unity 的逻辑通常基于时间差（deltaTime），帧率只影响画面流畅度，不影响世界运行速度。再加上泰拉瑞亚的网络同步、多人判定和大量历史代码也都默认所有客户端每秒更新 60 次，这使得修改 FPS 不只是一个渲染设置问题，而是会破坏整个游戏逻辑基础。

Q2:这个游戏到底是如何运作的？每个tick发生了什么？
A:泰拉瑞亚在底层运行时，游戏始终以固定的逻辑更新作为核心驱动，每秒稳定执行 60 次逻辑帧(tick)，在每一次逻辑帧中，游戏会按顺序处理玩家输入、更新玩家状态、更新所有 NPC 和弹幕的位置与行为、结算碰撞与伤害、推进世界时间并处理事件与网络同步，这一步决定了世界“实际发生了什么”；在逻辑更新完成之后，绘制系统才会读取当前已经确定的世界状态，把玩家、NPC、弹幕、光照和特效画到屏幕上，绘制帧只负责显示结果，不参与任何计算，也不会修改游戏数据，因此即使画面帧率波动或掉帧，世界仍然按照固定的 60 次逻辑更新稳定运行，这种将逻辑与绘制严格分离、并以固定逻辑帧作为时间基准的设计，构成了泰拉瑞亚游戏运行的底层基础。

Q3:泰拉瑞亚每一tick的逻辑执行，存在并行执行的代码吗？
A：泰拉瑞亚的逻辑帧执行基本上是严格顺序的单线程、从前到后处理，没有并行计算。在每一帧里，游戏会按固定顺序处理各类对象：先处理玩家输入和状态，然后按顺序处理每一个NPC，弹幕的逻辑更新，再处理掉落物、粒子、Buff、世界事件、网络同步等，每一项都是等上一步完成后才执行下一步，这保证了逻辑的一致性和 determinism（确定性），也方便多人模式同步。如果把这些步骤并行执行，就会出现状态读取竞争、碰撞判定不一致和同步错误，这也是泰拉瑞亚没有在逻辑帧中使用多线程的原因。

Q4:泰拉瑞亚的绘制是怎么运行的？存在图层吗？
A:泰拉瑞亚的绘制系统是完全线性、按顺序执行的，每一帧画面都会依次处理游戏中的各类元素，从背景、方块到玩家、NPC、弹幕、掉落物以及特效，最终叠加光照和前景效果。整个过程没有并行计算，每个元素都是在前一个元素绘制完成后才被渲染，游戏也没有真正的图层概念，所谓的“层次感”只是通过渲染顺序和对象位置来实现的，而不是独立的渲染层或多线程处理。
