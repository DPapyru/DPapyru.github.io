# 教学文章写作指南

这篇文章是当前项目给新贡献者的默认入口。

从现在开始，默认主线只收敛到两件事：

1. 会在 `/tml-ide/?workspace=markdown` 里写 Markdown 教程。
2. 需要动画时，再学 `使用网页特殊动画模块.md` 这一篇。

也就是说：不会 C#、没配本地 IDE、不了解 `ContentProjects`，都不会阻止你先把文章写出来并完成预览。

## 先记住当前主线

请把学习顺序压缩成下面两步：

1. 先学本篇，把文章写出来。
2. 再学 `使用网页特殊动画模块.md`，决定是否给文章加 `anim.ts` 动画。

下面这些内容先不算新人必学：

1. `ContentProjects` 本地工程
2. FX / Shader 编写
3. 本地桥接服务
4. 复杂 C# 示例维护

它们仍然有用，但应该放到“已经能正常写文并提交”之后。

## 你要交付的最小结果

一篇真正可进入评审的文章，最低满足下面 7 条：

1. 路径位于 `site/content/<分类>/` 下。
2. 有 YAML Front Matter，至少包含 `title` 与 `description`。
3. 正文至少能让读者看懂“目标、示例、小结”。
4. 在网页 IDE 中预览正常，没有明显断链或占位词。
5. 能在“统一提交”面板中收集到你的文章源码与相关资源。
6. 回到本地仓库执行 `npm run build`，把生成的 `site/content/config.json`、`site/assets/search-index.json`、`site/sitemap.xml` 等文件一起提交。
7. 再执行 `npm run check-generated`，确认没有漏提生成物，PR 不会因为脏产物失败。

如果你现在只能完成前 5 条，已经足够做出一份可预览草稿；但还不能把“浏览器里能预览 + 统一提交能收集改动”当成“已经可 review”。

## 第一步：打开网页 IDE

入口：`/tml-ide/?workspace=markdown`

打开后，先认三个区域：

1. 左栏 `文件目录`：新建、切换、定位文章文件。
2. 中栏编辑区：写 Markdown、切换预览、修改元数据。
3. 顶部工具栏：`预览`、`元数据`、`新标签预览`、`统一提交`。

默认建议：先不要管右侧和其它子能力，先把一篇文章从“空文件”写到“可预览”。

## 第二步：新建一篇 Markdown 文章

推荐操作：

1. 点击左栏 `新增文件`。
2. 在“快速创建”里选择 `Markdown (.md)`。
3. 目录填写相对 `site/content` 的分类路径，例如 `如何贡献`。
4. 文件名填写 `你的名字-文章标题.md`。

创建后，文件会立刻加入工作区并自动保存到浏览器本地缓存。

如果你暂时不知道文件里该写什么，直接点击工具栏里的 `插入模板`，先得到一个能改的草稿。

## 第三步：先把 Front Matter 补齐

最小模板如下：

```yaml
---
title: 文章标题
description: 一句话说明本文解决什么问题
topic: article-contribution
---
```

对新人来说，先把这三个字段写对最重要：

1. `title`：给读者看的标题。
2. `description`：一句话交代本文解决什么问题。
3. `topic`：当前贡献文档统一使用 `article-contribution`。

如果你愿意再补充一点信息，可以继续加：

```yaml
author: 你的名字
difficulty: beginner
time: 15分钟
```

工具栏里的 `元数据` 按钮可以帮助你核对这些字段，不必一开始就手记所有键名。

## 第四步：正文只先学最常用的 Markdown

先掌握下面 6 种就够写大多数教程：

1. 二级标题：`## 小节标题`
2. 列表：`1.` 或 `-`
3. 代码块：三反引号包起来
4. 普通链接：`[说明](./另一篇.md)`
5. 图片：`![说明](../../assets/imgs/example.png)`
6. 提示框：

```text
> [!NOTE]
> 这里写提醒内容
```

写作时有一个很实用的经验：先把正文组织成固定 4 段。

1. 这章要解决什么问题
2. 最小示例是什么
3. 每一步在做什么
4. 学完以后下一步看什么

这样写出来的文章，对新人最友好，也最方便 reviewer 快速检查结构。

## 第五步：边写边预览，不要等到最后

当前 IDE 有三种和预览有关的能力：

1. `预览`：在编辑区切换预览/可视化编辑。
2. `新标签预览`：用站点 viewer 单独打开当前文档。
3. `发布前自检`：扫描常见问题，例如占位词、旧语法、可疑链接。

推荐顺序：

1. 先在当前页点 `预览`，检查结构和块级排版。
2. 需要模拟读者视角时，再点 `新标签预览`。
3. 提交前点一次 `发布前自检`，把 `TODO`、`待补充`、旧语法清掉。

如果你使用的是段落、标题、列表、引用块，预览页里还能直接做少量可视化修改；复杂代码块和表格，仍然建议回源码编辑。

## 第六步：把文章交给 reviewer

文写完后，先不要急着离开页面。

推荐按下面顺序处理：

1. 点击 `统一提交`。
2. 先点 `收集改动`，确认新增或修改的文件都在列表里。
3. 补一个清楚的 PR 标题，例如 `docs: 补充某主题入门教程`。
4. 确认无误后再点 `提交全部`。

如果你还没准备好直接提交，也至少要确认“统一提交”面板能识别到你的文章文件，这说明本次改动链路是通的。

这里要特别记住：`统一提交` 只负责收集当前工作区里的源码文件，不会替你重跑仓库构建，也不会自动补齐 `config.json`、搜索索引或 `sitemap.xml`。所以它更适合确认“文章草稿是否收集正确”，不能单独作为“PR 已经能过 CI”的依据。

## 写给第一次贡献的人

如果你担心“我不会编程，能不能先写文”，答案是：可以。

在当前项目里，优先级是这样的：

1. 先把教程写清楚。
2. 再决定是否需要代码片段。
3. 最后才是动画、Shader、本地工程这些增强能力。

不要一开始就把自己卡在工具链上。

## 附录：提交前的本地检查

本篇主线虽然先从网页 IDE 开始，但只要你准备把 PR 送进评审，就必须回到本地仓库补这两条验证：

```bash
npm run build
npm run check-generated
```

用途分别是：

1. `build`：刷新结构索引、搜索索引、动画产物和站点构建结果，并产出 PR 里必须一并提交的生成文件。
2. `check-generated`：模拟 CI，检查生成文件是否与源码一致，避免 PR 因“少提了生成物”直接失败。

补充说明：仓库里仍有部分历史内容未完全整理；如果你额外运行 `npm run check-content`，看到与自己这次改动无关的旧报错，请在 PR 描述里注明，不必误判成“自己这篇写坏了”。

## 下一步

如果你的文章需要演示运动、交互或可视化过程，继续阅读：`使用网页特殊动画模块.md`。

如果你只是来补一篇普通教程，到这里就已经够用了。
