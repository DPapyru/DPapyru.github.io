# 使用网页特殊动画模块

本篇是贡献主线的第二课。

第一篇 `教学文章写作指南.md` 已经解决“如何把文章写出来”；这一篇只解决一件事：当你觉得纯文字不够时，怎样用 `anim.ts` 给文章补一个能预览的动态示意。

## 先定边界：这篇不要求你会什么

开始前，请先把范围收窄。

这篇不要求你先会：

1. 完整 TypeScript 语法
2. C# 项目开发
3. Shader / FX
4. 本地桥接服务
5. `ContentProjects` 本地工程

你现在只需要会“看懂一个小动画文件，改几个变量，并把它嵌进 Markdown 文章”。

## 什么时候需要 anim.ts

下面三种场景最适合补动画：

1. 纯文字不好说明运动轨迹。
2. 需要展示 `Time` 驱动的变化过程。
3. 需要让读者拖动、点击、观察交互反馈。

如果一张静态图就能讲清楚，不必强行上动画。

## 先补 4 个够用的 TypeScript 概念

当前项目里的 `anim.ts`，对贡献者来说，先理解下面 4 类概念就够了。

### 1. 变量和状态

例如：

```ts
this._mode = 0;
let phase = this._ctx.Time % 9;
```

你可以把它们理解成“动画当前记住的数据”。

1. `this._mode`：对象记住的状态。
2. `phase`：当前这一帧临时算出来的值。

### 2. 条件判断

例如：

```ts
if (this._ctx.Input.IsInside) {
    // 鼠标在画布内时执行
}
```

这类写法只是在表达“满足条件才执行”。动画里最常见的条件就是：

1. 输入是否进入画布
2. 当前时间落在哪一段
3. 当前模式是否锁定

### 3. 循环

例如：

```ts
while (i < 20) {
    i += 1;
}
```

循环的用途通常是：

1. 连续画多个点
2. 生成一串轨迹采样
3. 按顺序处理一组向量或颜色

### 4. 方法

例如：

```ts
DrawArrowRecipes(g, center, scale) {
    // 把一段可复用逻辑单独拎出来
}
```

当一段绘制逻辑变长时，把它拆成单独的方法，代码会更好改，也更不容易把主流程写乱。

## 你真正会频繁用到的上下文

在当前 IDE 里，编辑 `.anim.ts` 文件时，最值得先记住的是 `this._ctx` 下面这 4 个成员：

1. `Width`
2. `Height`
3. `Time`
4. `Input`

它们正好也是补全里最核心的一组提示。

可以这样理解：

1. `Width / Height`：画布有多大。
2. `Time`：动画已经走了多久。
3. `Input`：鼠标或交互状态。

如果你在编辑器里输入 `this._ctx.`，当前补全会优先把这些成员列出来。

## anim.ts 文件放哪里

约定很简单：

1. 源文件放在 `site/content/**/.anim.ts`。
2. 教程里最常见的是 `site/content/anims/*.anim.ts`。
3. 文章里引用的是 `.anim.ts` 路径，不是最终产物 `.js`。

最省心的创建方式：

1. 在 `/tml-ide/?workspace=markdown` 里点 `新增文件`。
2. 在“快速创建”里选择 `动画 TS (**/*.anim.ts)`。
3. 目录一般填 `anims` 或文章所在分类下的 `anims` 子目录。

## 从“改模板”开始，不要从空白文件开始

对新人最稳的路径，不是手写第一个动画文件，而是先复制一个已有模块再改。

建议优先参考这 3 个：

1. `anims/demo-basic.anim.ts`：最适合第一次确认嵌入链路。
2. `anims/vec2-basic-ops.anim.ts`：最适合理解 `Time`、`Input` 和状态切换。
3. `anims/animgeom-toolkit-recipes.anim.ts`：最适合理解循环、轨迹和辅助绘制。

建议第一轮只改这些东西：

1. 标题文字
2. 颜色
3. 速度
4. 一个参数或一种模式

这样最容易确认“我改动了什么，预览为什么变了”。

## 在文章里怎样引用动画

当前项目支持两种常用写法。

### 写法 1：协议链接

```text
[待补充说明](anims:anims/demo-basic.anim.ts)
```

### 写法 2：animts 代码块

````md
```animts
anims/demo-basic.anim.ts
```
````

什么时候选哪种：

1. 只想放一个嵌入入口，用协议链接就够。
2. 想让源码块和文章结构更统一，优先用 `animts` 代码块。

无论哪种写法，第一行最终都要落到真实的 `**/*.anim.ts` 路径上。

## 当前 IDE 里的最短工作流

推荐按下面顺序操作：

1. 打开你的 Markdown 文章。
2. 在文中插入 `anims:` 链接或 `animts` 代码块。
3. 再打开对应的 `.anim.ts` 文件。
4. 一边改动画文件，一边回文章页点 `预览` 观察结果。

当前行为可以这样理解：

1. 文章没有引用 `*.anim.ts` 时，状态会提示“未激活（当前文章未引用 *.anim.ts）”。
2. 文章引用了动画，但对应文件还没在工作区打开时，会提示“未激活（引用的 *.anim.ts 尚未在工作区打开）”。
3. 路径引用正确且文件已打开后，IDE 会对 `.anim.ts` 做防抖编译并同步到预览。

这意味着：你不需要先配额外服务，先把“文章引用路径 + 工作区里打开动画文件”这条链路走通就行。

## 给第一次写动画的人 3 个练习

### 练习 1：只改文字和颜色

目标：确认你能让预览产生“看得见的变化”。

优先改：

1. `g.Text(...)` 里的文案
2. `new Color(...)` 的颜色值

### 练习 2：只用 Time 做匀速变化

目标：理解动画不是“每帧都手写不同图”，而是“用时间算出当前状态”。

可以优先观察这类表达式：

```ts
let phase = this._ctx.Time % 9;
```

或者：

```ts
MathF.Sin(this._ctx.Time)
```

### 练习 3：只用 Input 做一个交互

目标：让动画对鼠标或模式切换产生反馈。

可以优先观察这些字段：

1. `this._ctx.Input.X`
2. `this._ctx.Input.Y`
3. `this._ctx.Input.IsInside`
4. `this._ctx.Input.Mode`

## 常见错误排查

### 1. 文章里写了路径，但没有动画

优先检查：

1. 路径是不是 `anims/.../*.anim.ts`
2. 路径是不是文章当前目录可解析到的真实文件
3. 对应 `.anim.ts` 文件是否已经在工作区打开

### 2. 能看到文章，但补全很少

请先确认你编辑的真的是 `.anim.ts` 文件。

在这个场景里，最先应该依赖的是 `this._ctx.` 这一组补全，而不是去背完整 API 名单。

### 3. 改了代码，预览没变

优先排查两件事：

1. 当前文章引用的是不是你正在编辑的那一个路径
2. 文章是否重新进入了 `预览` 状态

### 4. 一上来就觉得语法太多

先别追求“从零写一个完整动画”。先复制模板、改 1 个变量、看见 1 次变化，再继续下一步。
