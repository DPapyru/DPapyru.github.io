# 教学文章写作指南

这篇文章是 `site/content/怎么贡献/` 的入口文档。

目标只有一个：让新贡献者在当前项目中，用稳定流程写出可发布、可维护的教程。

如果你只想看语法速查，请直接跳到 `站点Markdown扩展语法说明.md`。

## 建议阅读顺序

1. 教学文章写作指南（本篇）
2. 站点Markdown扩展语法说明
3. 在线写作IDE使用教程
4. ContentProjects解决方案说明
5. 使用网页特殊动画模块

## 你要交付的最小结果

一篇可合并文章至少满足下面 6 条：

1. 路径位于 `site/content/<分类>/` 下
2. 有 YAML Front Matter，至少含 `title` 与 `description`
3. 正文有清晰结构，至少包含“目标、示例、小结”
4. 内容结构清晰，读者按顺序就能读懂
5. 本地运行 `npm run build` 成功
6. 本地运行 `npm run check-content` 成功

## 先选工作方式

### 方式A：网页 IDE 写作

适合：改文案、写新章节、快速提 PR。

入口：`/tml-ide/`

优点：

- 写作、预览在同一页
- 支持多文件编排并合并输出
- 可直接提交 PR

限制：

- 不用于真实 tModLoader 项目编译
- 不用于复杂 C# 调试

### 方式B：本地 IDE 写作

适合：需要 C# IntelliSense、跨文件重构、与本地 Mod 项目联动。

入口：`site/ContentProjects.sln`

详见：`ContentProjects解决方案说明.md`

## 文章目录与命名约定

推荐路径：`site/content/<分类>/作者-标题.md`

例如：

- `site/content/怎么贡献/YourName-某主题入门.md`
- `site/content/Modder入门学习/某子目录/YourName-某章节.md`

命名建议：

1. 作者名尽量使用英文或可稳定编码的字符
2. 文件名不要使用多余空格
3. 不要手工改生成文件作为最终结果

## Front Matter 规范

最小模板：

```yaml
---
title: 文章标题
description: 一句话说明本文解决什么问题
topic: article-contribution
---
```

推荐模板：

```yaml
---
title: 文章标题
author: 你的名字
date: 2026-02-07
last_updated: 2026-02-07
difficulty: beginner
time: 15分钟
description: 一句话说明本文解决什么问题
topic: article-contribution
order: 100
prev_chapter: 上一篇.md
next_chapter: 下一篇.md
min_c: 0
min_t: 0
---
```

说明：

- 分类来自目录，不需要在 YAML 里写 `category`
- `prev_chapter` 与 `next_chapter` 建议填同目录真实文件名
- 如果没有上下章，可省略字段，不要写 `null`

## 正文结构建议

推荐把每篇文章写成 6 段：

1. 本章目标
2. 前置条件
3. 最小可运行示例
4. 分步解释
5. 常见问题
6. 小结与下一步

这样做的好处：

- 读者知道先做什么再理解什么
- 维护者可以快速定位改动位置

## 给不会 C# 的新人的写作原则

项目现在不再做分流。为了照顾不会 C# 的新人，建议按下面方式组织内容：

1. 先讲概念和目标，再给完整可运行示例
2. 每个代码块后立刻解释关键字段含义
3. 在章节末尾放 1~2 个判断或单选题，确认是否理解

推荐在文中直接补充“术语解释行”，例如：

- `Item.damage`：武器基础伤害
- `ModItem`：自定义物品类型
- `SetDefaults`：设置物品默认属性的方法

## 当前项目的构建与校验命令

常用命令：

1. `npm run build`
2. `npm run check-content`
3. `npm run check-generated`

说明：

- `build` 会更新结构索引、搜索索引与动画产物
- `check-content` 会做内容规则校验
- `check-generated` 用于对齐 CI 行为

## 提交前自检清单

1. 文章元数据完整且语义清楚
2. 新人能按正文顺序读懂，不依赖额外背景
3. 相对链接和图片路径可打开
4. 文中命令与当前仓库脚本一致
5. 本地校验命令通过

## 常见失败场景

### 场景1：构建后出现大量变化

先确认是否改动了源内容，再决定是否提交生成文件。不要直接手改生成结果。

### 场景2：新人反馈看不懂

先补术语解释，再把大段代码拆成“步骤 + 对应代码”小节。

### 场景3：章节导航异常

检查 `prev_chapter` 与 `next_chapter` 是否指向同目录真实文件。
